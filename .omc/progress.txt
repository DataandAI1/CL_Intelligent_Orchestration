Issue 2: Dockerfile HEALTHCHECK marked cl-web unhealthy despite working HTTP
  - Symptom: every external check (curl -sI http://localhost:8088) returned HTTP/1.1 200 with correct body and headers, but `docker inspect` showed cl-web as unhealthy.
  - Diagnosis: `docker inspect cl-web` health log showed "wget: can't connect to remote host: Connection refused". Inside the container, `wget http://localhost:80/` failed but `wget http://127.0.0.1:80/` succeeded. nginx was listening on tcp 0.0.0.0:80 (IPv4 only).
  - Root cause: BusyBox wget in nginx:alpine resolves "localhost" to ::1 (IPv6) first. nginx.conf has only `listen 80;` (IPv4), no `listen [::]:80;`, so the IPv6 connection was refused.
  - Fix (one character): change Dockerfile HEALTHCHECK from `wget -qO- http://localhost:80/` to `wget -qO- http://127.0.0.1:80/`. After rebuild, cl-web reached healthy in ~12s.

Decisions / patterns:
  - Schema column for node-side schema text named `node_schema`, not `schema`, to avoid a PostgreSQL keyword/identifier conflict (Postgres parses `schema.` as a schema qualifier in some contexts). Wire shape still uses `schema` via mapping.
  - client_node_id / client_edge_id stored as TEXT, not UUID. The SPA generates ad-hoc string ids like `n-${Date.now()}-${rand}` (see DesignView.handleAddNode) — coercing to UUID would break round-trips of legacy clients and complicate the API contract. The (project_id, client_node_id) UNIQUE is the integrity guarantee.
  - Edge source/target stored as TEXT references to client_node_id (not FK references to design_nodes(id)). Reason: replace-strategy on PUT design wipes nodes then inserts; FKs would force ordering. The SPA's own consistency rules ensure source/target match a current node.
  - All routes use parameterized queries. Updates compose SET clauses dynamically by index ($1, $2, ...) — no string concat of user input.
  - `validate()` middleware narrows the request part using zod's safeParse and replaces the request body with the parsed value, so downstream handlers see coerced data (e.g., defaults applied for empty requirement arrays).
  - withTransaction() helper centralizes BEGIN/COMMIT/ROLLBACK; both requirements and design PUT routes use it.

  - Optimistic concurrency. Replace-strategy on PUT design/requirements works for the single-user scenario but is not race-safe.
  - File-uploaded asset storage (from StartPage's "Context Assets" upload). The text is analyzed via Gemini at upload time and never persisted as a raw asset.
  - Migration rollback path tested. `npm run migrate:down` exists in package.json but was not exercised in this iteration.
  - Built Docker images of the api service. The cl-api container was not run via docker compose in this iteration (the API was run on the host against the dockerized DB to keep verification fast). `docker compose config` validates the YAML; an end-to-end container run is a fast follow-up.
